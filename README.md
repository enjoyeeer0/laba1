## Теоретическое обоснование

Рефакторинг выполнен на основе пяти принципов SOLID.

SRP (Single Responsibility Principle): класс должен иметь одну ответственность. Если класс одновременно работает с оплатой, складом, уведомлениями и отчетами, это признак нарушения SRP.

OCP (Open/Closed Principle): система должна расширяться добавлением нового кода (новых реализаций), а не изменением уже проверенного. Большие if-else и switch обычно заменяются стратегиями.

LSP (Liskov Substitution Principle): дочерние типы должны заменять базовый без нарушения корректности. Если наследник не может безопасно выполнять базовый контракт, иерархия построена неверно.

ISP (Interface Segregation Principle): лучше несколько специализированных интерфейсов, чем один «толстый». Клиент не должен зависеть от методов, которые ему не нужны.

DIP (Dependency Inversion Principle): верхнеуровневые модули должны зависеть от абстракций, а не от деталей. Прямое создание зависимостей через new внутри бизнес-сервисов увеличивает связанность и ухудшает тестируемость.

Ожидаемый результат:

1. Снижение связанности между компонентами.
2. Рост расширяемости и тестопригодности.
3. Устранение архитектурных нарушений в целевых файлах.
